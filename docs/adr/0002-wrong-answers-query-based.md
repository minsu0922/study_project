# ADR-0002: 오답노트는 별도 테이블 없이 Submission 조회 기반으로 시작

- 상태: 채택됨
- 날짜: 2026-07-01
- 관련 로드맵: MVP / 로드맵 4(복습 추천)

## 맥락 (Context)
오답노트(`GET /api/me/wrong-answers`)가 필요하다. 구현 방식은 두 갈래:
전용 오답 테이블을 두고 채점 시 적재할지, 기존 `Submission` 데이터를 조회로 유도할지.

## 검토한 선택지 (Options)
1. **조회 기반** — `Submission where is_correct=false`를 문제 단위로 집계. 추가 테이블/쓰기경로 없음. 단, 집계 쿼리가 복잡·무거워질 수 있음.
2. **전용 `WrongNote` 테이블** — 채점 시 오답이면 upsert, 정답이면 삭제. 읽기 단순·빠름. 단, 쓰기 경로 복잡, 정합성 관리(중복/삭제) 부담, 복습 상태 없이는 과설계.

## 결정 (Decision)
**1안(조회 기반)** 채택. 오답노트는 Submission의 파생 뷰로 취급.
근거: MVP엔 "복습 상태(다음 복습일, 복습 횟수)"라는 독립 데이터가 아직 없다. 그게 없으면 전용 테이블은 Submission 중복일 뿐이다. 인덱스 `idx_submission_user_correct (user_id, is_correct, submitted_at)`로 조회 성능을 확보한다.

## 결과 (Consequences)
- 긍정적: 스키마·쓰기경로 단순. 채점 로직이 오답 테이블 관리와 결합되지 않음.
- 트레이드오프: "문제당 최신 오답 1행" 집계 쿼리 필요(GROUP BY / 윈도우). 데이터 커지면 비용↑.
- 재검토 트리거: **복습 추천(로드맵 4)** 도입 시. 망각곡선 기반 "다음 복습일·복습 횟수·상태(미복습/복습중/완료)"라는 **고유 상태**가 생기는 순간 → `ReviewItem` 테이블로 분리. 그때가 조회 파생 뷰의 한계이자 분리의 정당한 근거.

## 면접 대본 요약
"오답노트를 처음부터 테이블로 만들지 않았습니다. 초기엔 오답노트가 제출 이력의 파생 뷰에 불과해서, Submission을 조건 조회하고 (user_id, is_correct, submitted_at) 복합 인덱스로 성능을 확보했습니다. 전용 테이블은 '복습 상태'라는 고유 데이터가 생길 때, 즉 망각곡선 기반 복습 추천을 붙이는 시점에 ReviewItem으로 분리하기로 했습니다. 중복 데이터를 미리 만들지 않고 분리 시점을 데이터의 고유성으로 판단한 사례입니다."
